# 0.2.3 Migration Guide: Optimized Cache Structure

## Overview

Cache structure has been optimized to eliminate duplicate source tile downloads while maintaining per-geometry result caching.

**Problem:** The nested cache structure introduced in 0.1.0 duplicated raw source tiles (e.g., SRTM `.hgt` files) for every request, even when geometries overlapped.

**Solution:** Separate source tiles (global, reused) from result extraction (per-geometry, cached).

## Cache Structure Changes

### Before (0.2.2 - inefficient)
```
tiles/
  srtm30/
    {cache_key1}/
      hgt/          ← SRTM tiles (duplicated)
      gz/           ← SRTM tiles (duplicated)
      result.tif    ← extraction for geometry 1
      result_metadata.json
    {cache_key2}/
      hgt/          ← SRTM tiles (DUPLICATED AGAIN!)
      gz/           ← SRTM tiles (DUPLICATED AGAIN!)
      result.tif    ← extraction for geometry 2
      result_metadata.json
```

### After (0.2.3 - optimized)
```
tiles/
  srtm30/
    _tiles/                  ← Global source tile cache
      hgt/                   ← Downloaded once, reused
        N00E006.hgt
        N01E006.hgt
      gz/                    ← Downloaded once, reused
        N00E006.hgt.gz
        N01E006.hgt.gz
    {cache_key1}/            ← Result per geometry
      result.tif             ← extraction for geometry 1
      result_metadata.json
    {cache_key2}/            ← Different geometry
      result.tif             ← extraction for geometry 2
      result_metadata.json
```

## Benefits

- **50-90% storage savings** for overlapping/nearby extractions
- **Faster subsequent requests** to the same provider (no redownload)
- **Same result caching** per unique geometry (no regression)
- **Backward compatible** API — no changes to `extract_area()` or `get_result()`

## Breaking Changes

None for end-users. Internal structure changes only.

### For Custom Provider Implementations

If you have a custom provider, update `download_tiles()` accordingly:

**Before:**
```python
def download_tiles(self):
    output_dir = os.path.join(self._tile_directory, "hgt")
    os.makedirs(output_dir, exist_ok=True)
    # ... download to output_dir ...
    return [os.path.join(output_dir, f) for f in tiles]
```

**After:**
```python
def download_tiles(self):
    output_dir = os.path.join(self._source_tile_directory, "hgt")
    os.makedirs(output_dir, exist_ok=True)
    # ... download to output_dir (same as before) ...
    return [os.path.join(output_dir, f) for f in tiles]
```

The key: Use `self._source_tile_directory` instead of `self._tile_directory` for raw tiles.

### Path Properties (DTMProvider class)

Three key properties now define cache layout:

```python
# Global source tile directory (shared across all geometries)
self._source_tile_directory = os.path.join(directory, self._code, "_tiles")

# Provider-level directory (parent of both source tiles and result dirs)
self._provider_directory = os.path.join(directory, self._code)

# Result directory (specific to this request's geometry)
self._tile_directory = os.path.join(self._provider_directory, self._cache_key)
```

## Data Migration

Existing cached data in 0.2.2 format is NOT automatically migrated. To clean up:

```bash
# Delete old nested cache (will redownload on next request)
rm -r tiles/*/[a-f0-9]{24}/

# Keep source tiles if you want to preserve downloads
# (they were scattered; cleanup is recommended)
```

Or use fresh `tiles/` directory for 0.2.3.

## Detailed Changes

### 1. DTMProvider.__init__()

**Location:** [pydtmdl/base/dtm.py](pydtmdl/base/dtm.py#L184-L214)

```python
# OLD (0.2.2)
self._tile_directory = os.path.join(directory, self._code)
os.makedirs(self._tile_directory, exist_ok=True)
self._result_tiff_path = os.path.join(self._tile_directory, "result.tif")

# NEW (0.2.3)
self._provider_directory = os.path.join(directory, self._code)
self._source_tile_directory = os.path.join(self._provider_directory, "_tiles")
self._tile_directory = os.path.join(self._provider_directory, self._cache_key)
os.makedirs(self._source_tile_directory, exist_ok=True)
os.makedirs(self._tile_directory, exist_ok=True)
self._result_tiff_path = os.path.join(self._tile_directory, "result.tif")
```

**Impact:** All providers inherit this automatically. No changes needed unless custom.

### 2. Provider-Specific Updates

#### SRTM Provider

**Location:** [pydtmdl/providers/srtm.py](pydtmdl/providers/srtm.py#L19-L27)

**OLD (0.2.2):**
```python
def __init__(self, *args, **kwargs):
    super().__init__(*args, **kwargs)
    self.hgt_directory = os.path.join(self._tile_directory, "hgt")
    self.gz_directory = os.path.join(self._tile_directory, "gz")
```

**NEW (0.2.3):**
```python
def __init__(self, *args, **kwargs):
    super().__init__(*args, **kwargs)
    self.hgt_directory = os.path.join(self._source_tile_directory, "hgt")
    self.gz_directory = os.path.join(self._source_tile_directory, "gz")
```

This change applies to all providers that store raw tiles. Search your provider for `os.path.join(self._tile_directory, ...)` where raw tiles are downloaded.

**Providers affected:** All that download and cache tiles locally:
- SRTM30
- All WCS/WMS-based providers that cache intermediate .tif files
- Any custom provider that stores source tiles

**Providers unaffected:** Those that don't cache (e.g., pure APIs or streaming).

### 3. Cache Metadata (No Changes Required)

`result_metadata.json` remains in the result directory and includes:
- `cache_key` — geometry identifier (unchanged)
- `cache_path` — now points to `{provider}/{cache_key}/` (unchanged semantically)
- `cache_hit` — still indicates result.tif reuse (unchanged)

## Testing

After migration, verify:

1. **First extraction:** Downloads source tiles to `tiles/{provider}/_tiles/`
2. **Second overlapping extraction:** Reuses source tiles, no redownload
3. **Cache hit:** Returns result.tif instantly from `tiles/{provider}/{cache_key}/`
4. **Different geometry:** Reuses source tiles, creates new cache_key result

Example:
```python
from pydtmdl import DTMProvider

# Request 1: Center at (50.0, 10.0), width 4096m
result1 = DTMProvider.extract_area(
    center=(50.0, 10.0),
    width_m=4096,
    directory="tiles"
)
# Downloads SRTM tiles → tiles/srtm30/_tiles/
# Creates result → tiles/srtm30/{cache_key1}/result.tif

# Request 2: Center at (50.0001, 10.0001), width 4096m (overlaps!)
result2 = DTMProvider.extract_area(
    center=(50.0001, 10.0001),
    width_m=4096,
    directory="tiles"
)
# REUSES SRTM tiles from tiles/srtm30/_tiles/ ← KEY IMPROVEMENT
# Creates new result → tiles/srtm30/{cache_key2}/result.tif

print(result1.metadata.source_files)  # Lists reused tiles
print(result2.metadata.source_files)  # Same tiles, no redownload
```

## Backward Compatibility

- **Public API:** `extract_area()`, `get_result()`, result metadata — **unchanged**
- **Cache API:** `cache_key`, `cache_path` properties — **unchanged**
- **Config:** `directory` parameter — **unchanged**
- **Internal:** Provider path structure — **changed** (providers must use `_source_tile_directory`)

## Rollback

If you need to revert to 0.2.2 behavior:

```bash
git checkout 0.2.2
```

Note: You'll need to manually merge result caching logic if you want that feature retained.

## FAQ

**Q: Do I need to regenerate my cache?**  
A: No, but old nested structure won't be used. You can safely delete `tiles/*/[a-f0-9]{24}/` to free space.

**Q: Will my extractions be faster?**  
A: Yes, if you have overlapping/nearby requests. First request is the same speed; subsequent requests reuse tiles.

**Q: Can I move my old cache?**  
A: Old 0.2.2 cache is incompatible with 0.2.3. Fresh start recommended; downloads are cached globally now so recovery is fast.

**Q: Do I need to change my code?**  
A: No, unless you have a custom provider. Use `_source_tile_directory` for raw tiles instead of `_tile_directory`.
