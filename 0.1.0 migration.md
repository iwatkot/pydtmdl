# 0.1.0 Migration Guide

This release keeps the existing square-tile workflow working while adding the pieces needed for
backend and production usage.

## Summary

The main changes in `0.1.0` are:

- Rectangular ROI support with independent `width_m` and `height_m`
- Rotated ROI extraction with `rotation_deg`
- A higher-level extraction API for application code
- Structured extraction metadata instead of array-only workflows
- Explicit provider fallback support
- Stable cache keys and cache metadata
- Machine-readable error types for retry and fallback logic

## Backward Compatibility

Existing square-based usage continues to work:

```python
from pydtmdl import DTMProvider

provider_cls = DTMProvider.get_best(coords)
provider = provider_cls(coords, size=2048)
data = provider.image
```

What remains compatible:

- `size=` still works as before for square requests
- `provider.image` still returns the ROI as a NumPy array
- `provider.get_numpy()` still works for array-only consumers
- Existing provider selection flows using `get_best()` and `get_list()` still work

## New API Surface

### 1. Rectangular and Rotated Requests

Providers now accept optional geometry arguments in addition to the legacy `size` argument:

```python
provider = provider_cls(
    coords,
    width_m=4096,
    height_m=2048,
    rotation_deg=30,
)
```

Notes:

- If `height_m` is omitted, square behavior is preserved
- `size` remains the compatibility path for square requests
- Providers download enough source coverage for the rotated rectangle's bounding extent
- Final output is cropped to the requested rotated ROI, not the oversized download bounds

### 2. High-Level Extraction API

For backend and worker code, use the new application-facing helper:

```python
from pydtmdl import DTMProvider

result = DTMProvider.extract_area(
    center=coords,
    width_m=4096,
    height_m=2048,
    rotation_deg=30,
    provider_code="switzerland",
    fallback_provider_code="srtm30",
)
```

This is the recommended entry point when you need fallback, metadata, and cache awareness.

### 3. Structured Result Metadata

The new `get_result()` and `extract_area()` calls return a `DTMExtractionResult` with:

- `data`: the extracted NumPy array
- `metadata`: structured information about the extraction

Available metadata includes:

- requested provider
- actual provider used
- provider name
- resolution
- output path
- output CRS
- array shape
- dtype
- nodata value
- cache hit status
- cache key
- cache path
- fallback usage
- primary failure reason when fallback was used
- source files used
- requested center, width, height, and rotation

Example:

```python
result = provider.get_result(fallback_provider_code="srtm30")

print(result.metadata.cache_hit)
print(result.metadata.output_path)
print(result.metadata.actual_provider)
```

## Fallback Behavior

Fallback is now explicit and machine-readable.

```python
result = DTMProvider.extract_area(
    center=coords,
    width_m=4096,
    height_m=2048,
    provider_code="baden",
    fallback_provider_code="srtm30",
)
```

If the primary provider fails, the result metadata records:

- the requested provider
- the actual provider used
- whether fallback was used
- the primary failure reason and error type

## Machine-Readable Errors

Production code can now catch stable error classes instead of parsing generic exception strings.

Available error types include:

- `ProviderUnavailableError`
- `OutsideCoverageError`
- `AuthConfigMissingError`
- `DownloadFailedError`
- `ReprojectionFailedError`
- `CropExtractionError`

Each error can be serialized through `to_details()` for worker logs or API responses.

```python
from pydtmdl import DownloadFailedError

try:
    result = DTMProvider.extract_area(center=coords, width_m=4096)
except DownloadFailedError as exc:
    print(exc.to_details().model_dump())
```

## Cache Changes

Cache behavior is now geometry-aware.

Important changes:

- cache keys are stable for the same provider and geometry
- cache directories are now scoped by provider and request cache key
- the final extracted raster is cached as a reusable output artifact
- metadata is stored next to the cached output

If you previously inspected raw tile directories directly, account for the new nested cache layout.

## Provider-Specific Improvements

The following providers were updated so rectangular requests are covered correctly:

- `czech_dmr5g`
- `lithuania`
- `usgs_wcs`

These providers now request source data using dimensions that match the requested ROI instead of
forcing square-only output assumptions.

## Recommended Migration Path

If your code only needs a NumPy array and square requests, no changes are required.

If your code is part of a worker, API, or processing pipeline, migrate as follows:

1. Replace `size=` with `width_m=` and `height_m=` where rectangular extraction is needed.
2. Use `rotation_deg=` when the ROI is not axis-aligned.
3. Prefer `DTMProvider.extract_area(...)` or `provider.get_result()` instead of `provider.image`.
4. Use structured metadata for cache handling, output tracking, and fallback reporting.
5. Catch the new typed errors instead of relying on generic `ValueError` and `RuntimeError`.

## Dependency Update

`pyproj` is now an explicit dependency because ROI geometry and rotated extraction rely on local
coordinate transformations.

## Testing Coverage Added

This release also adds offline regression coverage for:

- rectangular rotated extraction
- stable cache keys and cache hits
- fallback reporting
- machine-readable failures

That coverage is intended to validate the new production-oriented flow without depending on live
provider availability.
