# 0.3.0 Migration Guide: Valid Coverage Threshold

## Overview

This release adds an optional `min_valid_coverage` argument to DTM and imagery extraction.

The new argument lets callers reject outputs that technically contain some data but still have too
many no-data holes. This is useful for production pipelines where a partially empty raster should
trigger fallback or fail clearly instead of being saved as a successful result.

## New API Surface

`min_valid_coverage` is available on both high-level extraction APIs:

```python
from pydtmdl import DTMProvider, ImageryProvider

dtm_result = DTMProvider.extract_area(
    center=(45.5, 10.0),
    width_m=4096,
    height_m=4096,
    provider_code="srtm30",
    min_valid_coverage=0.98,
)

imagery_result = ImageryProvider.extract_area(
    center=(39.10495723284634, -106.38702291276705),
    width_m=8192,
    height_m=8192,
    provider_code="naip",
    min_valid_coverage=0.98,
)
```

Values are fractions from `0.0` to `1.0`:

- `None` keeps the previous behavior and only rejects fully empty outputs
- `0.98` requires at least 98% valid pixels
- `1.0` requires complete valid coverage

## Behavior

The coverage check runs after the requested ROI has been cropped or resampled. It is applied to:

- freshly generated `result.tif` files
- cached `result.tif` files loaded from an existing cache entry
- primary providers and fallback providers

If coverage is below the requested threshold, extraction raises `CropExtractionError`:

```text
The output raster has insufficient valid coverage: 77.21% valid, required 98.00%.
```

## Fallback Interaction

Coverage failures participate in the existing fallback flow.

```python
result = ImageryProvider.extract_area(
    center=(39.10495723284634, -106.38702291276705),
    width_m=8192,
    height_m=8192,
    provider_code="naip",
    fallback_provider_code="sentinel2_l2a",
    min_valid_coverage=0.98,
)
```

If the primary provider returns an output below the threshold, pydtmdl raises a provider error
internally and tries the fallback provider. The same threshold is applied to the fallback output.

## Valid Pixel Rules

For DTM outputs, a pixel is valid when it is not masked as no-data by the raster.

For imagery outputs, a pixel is valid when:

- it is not masked as no-data
- not all color bands are zero

This catches imagery mosaics where missing pixels were written as black RGB `0,0,0`.

## Backward Compatibility

This is opt-in. Existing calls behave the same unless `min_valid_coverage` is provided.

```python
result = DTMProvider.extract_area(
    center=(45.5, 10.0),
    width_m=4096,
)
```

The call above keeps the old behavior.
