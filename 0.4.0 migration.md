# 0.4.0 Migration Guide: Application-Ready Asset Outputs

## Overview

This release adds an asset-focused workflow for applications that do not want to consume GeoTIFFs.

The new APIs let pydtmdl produce final JPG/PNG files directly:

- DTM full/export PNG
- DTM capped editor-preview PNG
- imagery capped editor-preview JPG

GeoTIFFs may still exist inside the pydtmdl cache as temporary or intermediate rasters, but they
are no longer part of the application-facing output contract for these APIs.

## Why This Exists

The older `extract_area()` APIs return NumPy arrays and cached `result.tif` outputs. That is useful
for scientific or custom processing workflows, but it is awkward for app workers that only need
ready-to-upload assets.

For large maps, the old downstream pattern was especially risky:

1. Ask pydtmdl for a GeoTIFF or NumPy array.
2. Reopen the GeoTIFF in the app worker.
3. Reproject and normalize it into an editor image.
4. Upload the final image.

The new workflow moves that final processing into pydtmdl. The app can now wait for final JPG/PNG
assets and metadata.

## New API Surface

### Provider-Based DTM Assets

```python
from pydtmdl import extract_project_dtm

result = extract_project_dtm(
    center=(48.137154, 11.576124),
    width_m=600,
    height_m=600,
    rotation_deg=0,
    provider_code="bavaria",
    directory="tiles",
    max_edge=8192,
)

print(result.full_output_path)
print(result.preview_output_path)
print(result.min, result.max)
```

This writes:

- `full_output_path`: full processed DTM PNG for export
- `preview_output_path`: capped processed DTM PNG for editor use

Both outputs are normalized `uint16` PNG files.

### Uploaded/Local DTM Assets

```python
from pydtmdl import extract_project_dtm_from_file

result = extract_project_dtm_from_file(
    image_path="uploads/source-dtm.tif",
    center=(48.137154, 11.576124),
    width_m=600,
    height_m=600,
    rotation_deg=15,
    directory="tiles",
    max_edge=8192,
)
```

The input is a local GeoTIFF, but the outputs are still PNG assets.

### Provider-Based Imagery Assets

```python
from pydtmdl import extract_project_imagery

result = extract_project_imagery(
    center=(48.137154, 11.576124),
    width_m=600,
    height_m=600,
    provider_code="sentinel2_l2a",
    directory="tiles",
    max_edge=8192,
)

print(result.preview_output_path)
```

This writes one capped JPEG preview.

### Uploaded/Local Imagery Assets

```python
from pydtmdl import extract_project_imagery_from_file

result = extract_project_imagery_from_file(
    image_path="uploads/source-imagery.tif",
    center=(48.137154, 11.576124),
    width_m=600,
    height_m=600,
    directory="tiles",
    max_edge=8192,
)
```

## DTM Output Behavior

DTM asset generation produces app-ready heightmap images, not raw elevation rasters.

The full DTM PNG is:

- reprojected into the project grid
- optionally rotated
- normalized to `uint16`
- intended for export-time use

The preview DTM PNG is:

- generated from the same project request
- capped by `max_edge` and optional `max_pixels`
- normalized with the same world-height min/max
- intended for editor use

Important metadata:

```python
result.full_output_path
result.full_shape
result.full_resolution

result.preview_output_path
result.preview_shape
result.preview_resolution

result.min
result.max
result.pixel_value_max  # 65535
```

Use `min`, `max`, and `pixel_value_max` to map PNG pixel values back to real heights.

## Preview Reuse When Full Fits

If the full DTM output already fits inside the requested preview cap, pydtmdl does not create a
duplicate preview PNG.

In that case:

```python
result.preview_output_path == result.full_output_path
result.preview_shape == result.full_shape
```

If the full DTM is larger than `max_edge` or `max_pixels`, pydtmdl writes a separate capped preview
PNG.

## Imagery Output Behavior

Imagery asset generation produces one editor-ready JPEG preview:

```python
result.preview_output_path
result.preview_shape
result.preview_resolution
result.preview_format      # "jpg"
result.preview_mime_type   # "image/jpeg"
```

Imagery is always capped by `max_edge` and optional `max_pixels`. There is no full-resolution imagery
asset in this workflow.

## Project Grid

The asset APIs write outputs into a project-local grid:

- centered on `center`
- sized by `width_m` and `height_m`
- rotated by `rotation_deg`
- using `destination_crs="auto-utm"` by default

`auto-utm` selects the UTM zone from the project center.

You can override this with any rasterio/pyproj CRS string:

```python
result = extract_project_dtm(
    center=(48.137154, 11.576124),
    width_m=600,
    destination_crs="EPSG:32632",
)
```

## Source Buffer

By default, pydtmdl expands the source request before generating project outputs:

```text
source_buffer_m = clamp(max(width_m, height_m) * 0.04, 128, 512)
```

The final PNG/JPEG assets still use the requested `width_m` and `height_m`. The buffer is only used
to make sure resampling and rotation have enough source coverage.

You can override it:

```python
result = extract_project_dtm(
    center=(48.137154, 11.576124),
    width_m=600,
    source_buffer_m=256,
)
```

## Result Metadata

The new result models are:

- `DTMAssetResult`
- `ImageryAssetResult`
- `SourceManifest`

They are exported from `pydtmdl`.

Common metadata includes:

- requested provider
- actual provider
- fallback status
- primary failure reason when fallback was used
- cache hit status
- cache key and cache path
- project center, width, height, and rotation
- source resolution
- source files
- source manifest

The `source_manifest` is intended for audit and future regeneration workflows. It records the source
request and provider context without requiring the app to store a raw GeoTIFF as an asset.

## GeoTIFFs Are Internal

The asset APIs expose JPG/PNG output paths only.

Internal `.tif` files may appear under the pydtmdl cache directory, for example:

```text
tiles/{provider}/{cache_key}/assets/*.tmp.tif
tiles/{provider}/{cache_key}/merged.tif
tiles/{provider}/{cache_key}/reprojected.tif
```

Applications should not upload or persist these files as user-facing assets. Treat them as pydtmdl
cache implementation details.

## Fallback Behavior

Fallback works the same way as the structured extraction APIs:

```python
result = extract_project_dtm(
    center=(48.137154, 11.576124),
    width_m=600,
    provider_code="bavaria",
    fallback_provider_code="srtm30",
)
```

If the primary provider fails, pydtmdl tries the fallback provider and records:

```python
result.fallback_used
result.primary_failure_reason
result.requested_provider
result.actual_provider
```

## Provider Class Wrappers

The asset workflow is also available from provider classes:

```python
from pydtmdl import DTMProvider, ImageryProvider

dtm = DTMProvider.extract_project_dtm(
    center=(48.137154, 11.576124),
    width_m=600,
    provider_code="bavaria",
)

imagery = ImageryProvider.extract_project_imagery(
    center=(48.137154, 11.576124),
    width_m=600,
    provider_code="sentinel2_l2a",
)
```

## Example Workflow Script

This release includes `asset-focused-workflow.py`.

Run it from the repository root:

```bash
python asset-focused-workflow.py
```

It demonstrates:

- Bavaria DTM full PNG
- Bavaria DTM capped preview PNG
- imagery preview JPG

The workflow tries Bavaria imagery first and falls back to Sentinel-2 if the Bavaria WMS service
does not return usable tiles.

Generated files are written under:

```text
asset-focused-output/
```

That directory is ignored by git.

## Backward Compatibility

No existing extraction API is removed.

These still work as before:

```python
DTMProvider.extract_area(...)
ImageryProvider.extract_area(...)
provider.get_result(...)
provider.get_numpy(...)
provider.image
```

Use the new asset APIs when your application wants final JPG/PNG assets. Use the existing extraction
APIs when you want NumPy arrays or lower-level raster processing control.

## Performance Notes

The asset workflow avoids returning full DTM arrays to the caller and writes normalized PNG output
from disk-backed intermediate rasters. This reduces pressure on application workers that only need
final assets.

One older internal hotspot remains: multi-tile provider merging can still materialize a full mosaic
in memory. A future optimization should replace that with a more fully windowed or VRT-based path.

## Recommended Migration Path

For app workers:

1. Replace DTM `extract_area()` plus app-side GeoTIFF reopening with `extract_project_dtm(...)`.
2. Upload `result.full_output_path` for export use.
3. Upload `result.preview_output_path` for editor use.
4. If both paths are equal, store one asset and reference it for both roles.
5. Store `result.min`, `result.max`, and `result.pixel_value_max` with the DTM asset metadata.
6. Replace imagery `extract_area()` plus app-side processing with `extract_project_imagery(...)`.
7. Store `result.source_manifest` for audit or future regeneration.
